## Summary

This phase is a clean-room rewrite of the tool registration layer and agent factory. The primary technical goal is to eliminate all workarounds accumulated during Phase 3 (`_impl` renaming, `StructuredTool` aliases, `TYPE_CHECKING` Config wrapper, `retrieval_wrappers.py` counter logic) and replace them with the straightforward pattern that `deepagents 0.4.11` actually supports.

The key finding from reading the installed library source is that `create_deep_agent` passes `tools` directly to LangChain's `create_agent`, which passes them to `ToolNode`. `ToolNode` (langgraph) accepts `Sequence[BaseTool | Callable]` and automatically converts plain Python functions to `BaseTool` via `create_tool()` (the `@tool` decorator). This means **plain Python functions with proper type annotations and docstrings register as native tools with no wrapper needed**. The `@tool("name")` decorator was only ever needed to set a custom tool name — and that name can now be set via a `@tool(name="...")` kwarg, a `@tool("name")` positional call, or simply by relying on the function name.

The `get_model()` factory in `model_adapter.py` is already model-agnostic and well-structured. It accepts `Config` (or reads from env). The decision to rename the parameter to `model_id: str` directly (instead of `config`) is the cleanest path since `route_files` currently creates a `Config`-dependency problem. Removing the `Config` parameter from all tool signatures — and instead reading `CORPUS_DIR` from the environment at module level — eliminates both the `TYPE_CHECKING` workaround and the thin `_route_files_agent` wrapper.

**Primary recommendation:** In every tool file, remove the `@tool`-decorated alias and the `_impl` suffix. Define the function with the final public name directly. Use a module-level constant for `CORPUS_DIR` (read from env via `os.environ.get` at import time). Pass plain functions or `@tool`-decorated functions to `create_deep_agent`. Delete `retrieval_wrappers.py`. Rewrite tests to call tools via `.run()` / `.invoke()` on the `@tool`-decorated objects (which is the existing test pattern for `calculate` / `pct_change`).

### Pattern 1: Native Tool Registration (No _impl, No StructuredTool)

**What:** Define the tool function once with its public name. Decorate with `@tool` (optionally with an explicit name). Pass the decorated object directly to `create_deep_agent(tools=[...])`.

**When to use:** All tools in this codebase.

**Current pattern (to be removed):**
```python
# BEFORE — two-function pattern
def _calculate_impl(expr: str) -> dict:
    ...

calculate = tool("calculate")(_calculate_impl)
```

**Target pattern:**
```python
# AFTER — single function, directly decorated
from langchain_core.tools import tool

@tool
def calculate(expr: str) -> dict:
    """Evaluate an arithmetic expression safely using an AST whitelist. ..."""
    ...
```

Or equivalently (if name must differ from function name):
```python
@tool("calculate")
def calculate(expr: str) -> dict:
    ...
```

Both forms are supported. The `@tool` decorator converts the function to a `StructuredTool` (`BaseTool` subclass) whose `.run()` and `.invoke()` are exactly what tests currently use.

**Source:** langgraph `tool_node.py` lines 773-778, langchain `agents/factory.py` line 675, deepagents `graph.py` line 83.

### Pattern 2: Corpus Root — Module-Level Constant (Eliminates Config Wrapper)

**What:** Read `CORPUS_DIR` from the environment once at module import time. Store as a module-level constant. Tool functions reference the constant directly.

**Why this solves the problem:** The existing `_route_files_agent` thin wrapper existed because `_route_files_impl` accepted a `Config` parameter that Pydantic introspected as part of the tool schema, causing errors. Removing `Config` from the tool function signature entirely — and reading the path from env or a module-level constant — eliminates the need for a wrapper.

**Target pattern for `route_files.py`:**
```python
import os
from pathlib import Path

# Read corpus dir once at import time — no Config dependency in tool signature
_CORPUS_DIR = Path(os.environ.get("CORPUS_DIR", "../corpus/transformed")).resolve()

@tool
def route_files(question: str) -> dict:
    """Extract year references from question and return matching bulletin file paths. ..."""
    # Use _CORPUS_DIR directly — no config param needed
    corpus_dir = _CORPUS_DIR
    ...
```

**Tradeoff vs env var approach:** A module-level constant means the env var is read at import time, not at call time. This is fine for the production case (env var is set before any import). For tests, `_CORPUS_DIR` can be monkey-patched or the env var can be set before import. This is simpler than the `Config` parameter approach and avoids Pydantic schema introspection entirely.

**Alternative — keep Config but use lazy import inside function body:** If corpus path must remain dynamic, keep `get_config()` as a lazy import inside the function body (not as a parameter). This also avoids Pydantic introspection since `Config` never appears in the function signature.

```python
@tool
def route_files(question: str) -> dict:
    """..."""
    from src.config import get_config
    config = get_config()
    corpus_dir = Path(config.corpus_dir)
    ...
```

**Recommendation:** Module-level constant is the cleanest approach since `CORPUS_DIR` is a deployment-time value. The lazy import inside function body is acceptable as a fallback if test isolation requires it.

### Pattern 3: Model Factory — Simplified Signature

**What:** `get_model()` in `model_adapter.py` currently takes `Config | None`. The decision says: keep `get_model(model_id)` as a standalone factory. Two options for calling it:

Option A — Read model_id inside get_model from env (zero-arg form):
```python
def get_model() -> BaseChatModel:
    model_id = os.environ.get("MODEL_ID", "claude-sonnet-4-6")
    ...
```

Option B — Keep Config param for testability:
```python
def get_model(config: Config | None = None) -> BaseChatModel:
    if config is None:
        from src.config import get_config
        config = get_config()
    model_id = config.model_id
    ...
```

The CONTEXT.md says "Keep a standalone `get_model(model_id)` factory function" — this suggests Option A with an explicit `model_id: str` parameter. `agent.py` would call it as `get_model(os.environ.get("MODEL_ID", "claude-sonnet-4-6"))`.

**Recommendation:** `get_model(model_id: str)` — explicit string parameter, no Config dependency. `agent.py` reads `MODEL_ID` from env and passes to `get_model`. This makes `get_model` fully testable without any env setup.

### Pattern 4: create_deep_agent — Clean Factory

**What:** `create_agent` in `agent.py` becomes `create_deep_agent` with no tool import list — middleware handles discovery.

**Current code in `agent.py`:**
```python
agent = create_deep_agent(
    model=model,
    tools=[counted_rf, counted_sif, extract_table_block, calculate, pct_change,
           sum_values, normalize_answer],
    system_prompt=SYSTEM_PROMPT,
    backend=FilesystemBackend(root_dir="./scratch", virtual_mode=False),
    checkpointer=MemorySaver(),

```

**Target pattern:**
```python
# Import tools at top of agent.py (or inside create_agent)
from src.tools.route_files import route_files
from src.tools.search_in_file import search_in_file
from src.tools.extract_table_block import extract_table_block
from src.tools.calculate import calculate, pct_change, sum_values

def create_agent() -> CompiledStateGraph:
    model = get_model(os.environ.get("MODEL_ID", "claude-sonnet-4-6"))
    return create_deep_agent(
        model=model,
        tools=[route_files, search_in_file, extract_table_block,
               calculate, pct_change, sum_values, normalize_answer],

        checkpointer=MemorySaver(),
    )
```

Note: "auto-discovers native tools" in the CONTEXT.md means no retrieval_wrappers counters or indirect imports — the tool list is explicit but clean (no wrappers, no aliases).

### Pattern 5: Test Interface — .run() and .invoke()

**What:** After removing `_impl` suffixes, tests that called `_calculate_impl(...)` directly must be rewritten to call via the tool interface.

**Existing test pattern (already correct for `calculate`):**
```python
r = calculate.run("2 + 3")                    # single-arg tools
r = pct_change.invoke({"old": 100, "new": 200})  # multi-arg tools
```

**For tools where `_impl` tests verified invalid inputs:**
```python
# Tests that called _pct_change_impl("abc", 100) for INVALID_INPUT:
# Rewrite to call the underlying logic directly (keep private helper) OR
# call via .invoke() with string inputs — Pydantic will either coerce or error
r = pct_change.invoke({"old": "abc", "new": 100})
```

The key insight: if the `_impl` functions are removed entirely, tests for invalid-type inputs must call via `.invoke()`. Pydantic V2 validates before the function runs, so if the schema rejects the type (e.g. string where float expected), the test must expect a Pydantic `ValidationError`, not an `INVALID_INPUT` dict. If the function accepts `str | float | int` (or `Any`) for `old` and `new`, then `.invoke()` passes through and the function's own type conversion produces `INVALID_INPUT`.

**Recommendation (Claude's Discretion):** Keep a thin private helper (e.g. `_pct_change_calc`) that contains the actual logic, with the `@tool`-decorated public function calling it. This allows tests that need to pass deliberately wrong types to call the helper directly. The difference from the current pattern is that the helper is not the primary implementation unit — the `@tool`-decorated function IS the tool, and the helper is a testability aid. Alternatively, widen the type annotations on `old`/`new` to `float | str` and call via `.invoke()` for all tests.

### Anti-Patterns to Avoid

- **`TYPE_CHECKING` imports in function signatures:** Never import `Config` under `TYPE_CHECKING` for use in a tool parameter type. Pydantic introspects the actual type at schema build time and will hit `NameError` if it is unavailable.
- **Positional `tool("name")` on a function that tests call with positional args:** The existing bug was that `@tool("name")` was required as a positional argument name alias. The clean pattern eliminates this by using `@tool` with the function name as the tool name, or `@tool("name")` with keyword syntax.
- **Counter wrappers that shadow tool names:** `make_counted_route_files()` returned a new `StructuredTool` named `route_files`. This shadowed the real tool and created two distinct registration paths. The clean pattern has exactly one `route_files` object.
- **Importing `_impl` from tool modules in tests:** After the refactor, `_impl` functions may not exist. Tests should import and call the public `@tool`-decorated names.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Max agent turns / retrieval limit | Custom counter-wrapper per tool | deepagents native recursion_limit (1000) or LangGraph max_iterations | deepagents already sets `recursion_limit: 1000` in `.with_config()`; additional turn limits can be set via `ToolCallLimitMiddleware` from langchain if needed |
| Tool schema generation | Manual JSON schema dicts | `@tool` decorator (auto-infers from type annotations + docstring) | LangChain `@tool` handles Python type → JSON Schema mapping including `Optional`, `list`, `dict` |
| Model dispatch | `if model_id.startswith(...)` in multiple places | Single `get_model(model_id)` factory in `model_adapter.py` | Already done; keep centralized |

**Key insight:** The retrieval counter (`RETRIEVAL_EXHAUSTED`) was a workaround for a missing native feature that deepagents already provides via its `recursion_limit: 1000` config. The system prompt instruction "When route_files or search_in_file returns RETRIEVAL_EXHAUSTED, STOP calling that tool" will be removed. The agent naturally stops when no more tool calls are appropriate.

## Common Pitfalls

### Pitfall 1: Pydantic Schema Introspection of TYPE_CHECKING Imports

**What goes wrong:** If a function parameter type annotation references a class that is imported under `TYPE_CHECKING`, Pydantic raises `NameError` when building the tool's JSON Schema. This is what caused the original `Config` problem.

**Why it happens:** `TYPE_CHECKING` is `False` at runtime, so the import block is never executed. The annotation is a string reference at runtime, but Pydantic V2 evaluates it via `get_type_hints()` which resolves string annotations in the module's `__globals__`, where `Config` does not exist.

**How to avoid:** Never put a class in a tool function's signature if it's imported under `TYPE_CHECKING`. Options: (a) module-level constant, (b) lazy import inside the function body, (c) remove the parameter entirely.

**Warning signs:** `NameError: name 'Config' is not defined` during tool schema build, or `pydantic.errors.PydanticSchemaGenerationError`.

### Pitfall 2: Tool Name Collision When Two Objects Share a Name

**What goes wrong:** If `route_files` (the `@tool`-decorated function in `route_files.py`) and `counted_route_files` (from `retrieval_wrappers.py`) are both registered with `create_deep_agent`, or if the same name appears twice in the tools list, the agent sees a duplicate name and one silently shadows the other.

**Why it happens:** `ToolNode` builds a `dict[str, BaseTool]` keyed by name. A second tool with the same name overwrites the first.

**How to avoid:** After removing `retrieval_wrappers.py`, verify there is exactly one object named `route_files` and one named `search_in_file` across all imports.

**Warning signs:** Agent calls a tool but gets an unexpected response format, or a tool that should be limited is called with a different counter than expected.

### Pitfall 3: Optional[str] Parameter Rejection by Pydantic V2

**What goes wrong:** Pydantic V2 rejects `None` for a parameter typed `str = None` (old Pydantic V1 syntax). If `pct_change` has `unit_old: str = None`, Pydantic V2 raises `ValidationError` when `.invoke({"old": 100, "new": 200})` is called (because `unit_old` is omitted).

**Why it happens:** Pydantic V2 treats `str = None` as "required str with a wrong default". The correct V2 syntax is `unit_old: Optional[str] = None`.

**How to avoid:** All optional string parameters must use `Optional[str]` (from `typing`) or `str | None` syntax with a `= None` default.

**Warning signs:** `ValidationError: ... field required` or `... none is not an allowed value` when invoking a tool without optional params.

**Current status:** `_pct_change_impl` already uses `Optional[str]`. This must be preserved in the rewrite.

### Pitfall 4: Model Adapter — ChatAnthropicVertex Location

**What goes wrong:** `ChatAnthropicVertex` is imported from `langchain_google_vertexai.model_garden`, not from `langchain_anthropic`. Using the wrong import raises `ImportError` or produces a model that does not work with Vertex AI auth.

**Why it happens:** Claude on Vertex AI uses Google's auth infrastructure, not Anthropic's API key. The `langchain-google-vertexai[anthropic]` package provides the bridge.

**How to avoid:** Keep the existing import path: `from langchain_google_vertexai.model_garden import ChatAnthropicVertex`. Do not change it.

**Warning signs:** `ImportError`, or authentication errors when running Claude through Vertex AI.

### Pitfall 5: SYSTEM_PROMPT References RETRIEVAL_EXHAUSTED

**What goes wrong:** The current `SYSTEM_PROMPT` contains instructions about `RETRIEVAL_EXHAUSTED` — telling the agent to stop when it receives that error. After removing `retrieval_wrappers.py`, the agent will never see `RETRIEVAL_EXHAUSTED`. If the old system prompt text remains, the agent receives confusing instructions about an error it will never encounter.

**How to avoid:** Rewrite `SYSTEM_PROMPT` to remove all references to `RETRIEVAL_EXHAUSTED` and `RETRIEVAL_LIMIT`. The planning gate instruction (`write_todos` before retrieval) must be preserved verbatim since tests assert on it.

### Pitfall 6: run_question Config Parameter Removal

**What goes wrong:** `run_question(uid, question, config)` currently accepts a `Config` parameter. If tests pass a `Config` with a specific `corpus_dir` for routing, removing the `config` parameter breaks those tests.

**Why it matters:** Looking at `test_route_files.py`, tests call `_route_files_impl(question, config)` directly with a test config. After the refactor, those tests must pass corpus_dir via env var or another mechanism.

**How to avoid:** For unit tests of `route_files`, set `os.environ["CORPUS_DIR"]` before importing the module (or monkeypatch the module-level constant `_CORPUS_DIR`). The `run_question` signature can drop the `config` parameter — it now reads `MODEL_ID` from env, and corpus path is read at tool import time.

### Pitfall 7: test_agent.py Imports run_question_with_messages

**What goes wrong:** `test_agent.py` imports and uses `run_question_with_messages`. If that function is removed during the refactor, the integration test file breaks.

**How to avoid:** Preserve `run_question_with_messages` in `agent.py` (or rewrite it in a compatible form). It is needed for the "write_todos before retrieval" ordering assertion.

## Open Questions

1. **Private helpers for unit tests calling invalid types**
   - What we know: Current `test_calculate.py` calls `_pct_change_impl("abc", 100)` directly to test `INVALID_INPUT`. After removing `_impl`, calling `pct_change.invoke({"old": "abc", "new": 100})` may hit Pydantic validation before the function, changing the error type.
   - What's unclear: Whether Pydantic V2 will coerce `"abc"` to `float` (it won't — it will raise `ValidationError`) or pass it through (only if the type annotation is `Any`/`str`).
   - Recommendation: Verify by inspection. `_pct_change_impl` has `old: float` — Pydantic V2 will reject `"abc"`. Two clean solutions: (a) change `old: float` to `old: float | str | int` (keep existing coercion logic in function body), or (b) keep `_pct_change_impl` as a thin private helper that tests call directly. Option (a) is cleaner.

2. **corpus_dir in test_route_files.py**
   - What we know: Tests call `_route_files_impl(question, config)` where `config.corpus_dir` is set to the real corpus path. After removing the `config` parameter, the module-level `_CORPUS_DIR` will be read from `CORPUS_DIR` env var at import time.
   - What's unclear: Whether existing test infra sets `CORPUS_DIR` before running `test_route_files.py`.
   - Recommendation: Use `monkeypatch.setattr(route_files_module, "_CORPUS_DIR", CORPUS_DIR)` in tests that need a specific corpus path. This is the standard pytest pattern for module-level constants.

3. **Whether model-switch test asserts exact answer equality**
   - What we know: The CONTEXT.md says "assert same answer". For a simple factual lookup question (Q1), both models should return the same numeric value. But LLM answers can vary in phrasing.
   - What's unclear: Should the assertion be `==` (exact string match) or a looser check (both contain the correct number)?
   - Recommendation: For the planning phase, design the test with `==` but document that it may need to be loosened to numeric extraction if phrasing varies. Keep Q1 (calendar year 1940 defense expenditures) as the test question since it has a single unambiguous numeric answer.
