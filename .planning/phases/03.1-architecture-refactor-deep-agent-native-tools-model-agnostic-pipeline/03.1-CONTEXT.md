# Phase 3.1: Architecture Refactor ŌĆö Deep Agent Native Tools + Model-Agnostic Pipeline - Context

**Gathered:** 2026-03-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Refactor the tool registration and agent wiring so all Phase 1+2 tools run as native Deep Agent tools, and the pipeline works identically with any Gemini or Claude model on Vertex AI selected via MODEL_ID. No new capabilities ŌĆö this phase removes legacy workarounds and establishes the clean architecture that subsequent phases build on.

</domain>

<decisions>
## Implementation Decisions

### Tool migration pattern
- Eliminate the `_impl` / `StructuredTool` wrapper pattern entirely ŌĆö clean rewrite, delete what's no longer needed
- Native Deep Agent tool registration replaces LangChain `@tool` decorator and `StructuredTool` aliases
- `create_deep_agent` auto-discovers native tools ŌĆö no explicit tool import list in `agent.py`
- Drop `retrieval_wrappers.py` and the call-counting / `RETRIEVAL_EXHAUSTED` mechanism entirely ŌĆö use Deep Agents' native max-turns limit instead

### Model-agnostic boundary
- Support Gemini models and Claude models via Vertex AI (currently `gemini-2.0-flash` and `claude-sonnet-4-6`)
- Model selected via `MODEL_ID` environment variable only ŌĆö no runtime argument to `run_question`
- Keep a standalone `get_model(model_id)` factory function that maps `MODEL_ID` to the correct Vertex AI chat model instance ŌĆö `agent.py` calls this and passes the result to `create_deep_agent`
- Rewrite `SYSTEM_PROMPT` as part of this phase to match new native tool names and call patterns

### Config / tool signatures
- Remove the `Config` / `TYPE_CHECKING` import pattern that was causing Pydantic schema errors
- Claude's Discretion: choose cleanest approach for corpus root config (env var or module-level constant) ŌĆö eliminate the need for thin wrapper functions entirely

### Test strategy
- All tests can be rewritten to match the new architecture ŌĆö no obligation to preserve old call sites
- Tool unit tests call tools via `.run()` / `.invoke()` (native Deep Agent interface) instead of calling `_impl` directly
- Existing 3 integration smoke questions must still pass
- Add one new model-switching integration test: same question run with both `MODEL_ID=gemini-2.0-flash` and `MODEL_ID=claude-sonnet-4-6`, assert same answer

### Migration approach
- Clean rewrite of affected files ŌĆö rewrite functions, delete obsolete code, no incremental patching
- Update `STATE.md` and `ROADMAP.md` to remove stale architecture decisions (e.g., `_impl`, `StructuredTool`, retrieval counter workarounds)

### Definition of done
All four must be true:
1. `create_deep_agent` runs successfully with both Gemini and Claude via `MODEL_ID`
2. All 3 smoke questions pass on both models
3. All tests pass (rewritten as needed) including the new model-switch test
4. No legacy patterns remain: no `_impl`, no `StructuredTool`, no retrieval counter wrappers

### Claude's Discretion
- Whether `_impl` functions are kept as thin private functions for unit-testability, or removed entirely ŌĆö pick the cleanest approach
- Corpus root config mechanism (env var vs module-level constant) ŌĆö whatever eliminates the Pydantic schema introspection problem cleanest
- Exact system prompt wording ŌĆö rewrite to match native tool names/signatures

</decisions>
