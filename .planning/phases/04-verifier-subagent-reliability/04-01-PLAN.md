  <name>Task 1: Create verifier.py with subagent spec, system prompt, and helper functions</name>
  <files>workspace/src/tools/verifier.py</files>
  <action>
Create `workspace/src/tools/verifier.py` with these components:

1. **`_generate_token(answer: str) -> str`** — Private function. Uses `hashlib.sha256(answer.encode("utf-8")).hexdigest()[:16]` to produce a deterministic 16-char hex token. Import hashlib from stdlib.

2. **`resolve_era_column_header(target_series: str, candidate_headers: list[str], cutoff: float = 0.6) -> str | None`** — Public function. Uses `difflib.get_close_matches(target_series, candidate_headers, n=1, cutoff=cutoff)`. Returns the best match or None. Per user decision: fuzzy match at runtime, no hard-coded variant table. Activation: multi-era questions only (date range crosses era boundary).

3. **`VERIFIER_SYSTEM_PROMPT: str`** — Multi-line string constant. The verifier system prompt must:
   - Instruct the verifier that the task description contains the UID and proposed answer (do NOT use `{uid}` format placeholders in the system prompt itself — Pitfall 3 from research)
   - Define four checks:
     - Check 1: Evidence Coverage (HARD VETO) — every numeric value in extracted_values.txt must appear as a literal in evidence.txt. When exact column name matching fails, apply fuzzy matching via the era-aware resolver pattern (instruct the LLM to consider variant series names).
     - Check 2: Unit Consistency (HARD VETO) — all values input to the same calculation must carry the same unit annotation
     - Check 3: Arithmetic Re-execution (HARD VETO when calc.txt exists and is parseable) — re-execute each expression in calc.txt using the `calculate` tool; FAIL if re-executed result differs from recorded result by any amount (exact Decimal match per user decision). SKIP if calc.txt absent/unparseable AND answer type does not require arithmetic (Claude's discretion per user decision).
     - Check 4: Format Match (SOFT WARNING) — proposed answer must match normalizer format patterns; WARN but do not FAIL on cosmetic issues
   - Specify output format: Return ONLY a JSON object `{"status": "PASS"|"FAIL"|"ERROR", "issues": [...], "token": "<hex>" or null}`. Token is non-null ONLY on PASS, generated as sha256 hex prefix of the proposed answer.
   - Instruct: read scratch files using read_file tool (inherited from FilesystemMiddleware)
   - Instruct: for verification.txt audit trail, use the read-then-write append pattern (read_file current content, concatenate new attempt record, write_file combined content). Per Pitfall 4: no native append mode in FilesystemMiddleware.

4. **`VERIFIER_SUBAGENT_SPEC: dict`** — The SubAgent TypedDict-compatible dict:
   ```python
   VERIFIER_SUBAGENT_SPEC = {
       "name": "verifier",
       "description": "Stateless verification subagent. Pass the proposed answer and scratch UID. Returns status (PASS/FAIL/ERROR), issues list, and token.",
       "system_prompt": VERIFIER_SYSTEM_PROMPT,
       "tools": [calculate],  # explicit — arithmetic re-execution only; prevents inheriting domain tools (Pitfall 1)
   }
   ```
   Import `calculate` from `src.tools.calculate`. Pass `"tools": [calculate]` explicitly so the verifier gets only `calculate` plus filesystem tools from the default middleware stack. Do NOT omit the tools key (would inherit all main agent tools per Pitfall 1).

Module docstring should explain purpose: verification subagent spec and helper functions for Phase 4 reliability layer.

  <files>workspace/src/agent.py</files>
  <action>
Modify `workspace/src/agent.py`:

1. **Update `create_agent()`** — Add verifier subagent registration:
   - Add import: `from src.tools.verifier import VERIFIER_SUBAGENT_SPEC`
   - Add `subagents=[VERIFIER_SUBAGENT_SPEC]` parameter to the `create_deep_agent()` call, right after `tools=[...]`

2. **Update `SYSTEM_PROMPT`** — Add a new section `## Verification and Retry Protocol` at the end. This section must contain:

   ```
   ## Verification and Retry Protocol

   Before calling normalize_answer, you MUST call the verifier subagent:
     task(subagent_type="verifier", description="Verify answer for UID <uid>. Proposed answer: '<answer>'. Scratch directory: <uid>/. Check evidence coverage, unit consistency, arithmetic, and format. Return JSON with status, issues, token.")

   The verifier returns a JSON object with status, issues, and token.

   On PASS:
     - Use the returned token as the verification_token argument to normalize_answer.
     - Append the PASS result to verification.txt using the read-then-write pattern:
       1. read_file("{uid}/verification.txt") to get current content (may be empty)
       2. Concatenate: old_content + new attempt record
       3. write_file("{uid}/verification.txt", combined_content)
     - Format: "Attempt N: Status: PASS | Checks: evidence: PASS, arithmetic: PASS, units: PASS, format: PASS | Token: <token>\n---\n"

   On FAIL:
     - Read the issues list to understand what failed.
     - Perform targeted re-retrieval based on issues (e.g., unit FAIL -> re-retrieve the table with unit annotation; arithmetic FAIL -> re-check calc.txt expression).
     - Append the FAIL result to verification.txt using the same read-then-write pattern.
     - Retry the full answer derivation and call the verifier again.
     - You have 3 total attempts (1 original + 2 retries).

   On ERROR (verifier crashed/timed out):
     - Count this as a failed attempt. Same retry logic as FAIL.
     - Append the ERROR result to verification.txt.

   After 3 failed attempts:
     - Respond with EXACTLY: "cannot determine: [last verifier issues list]"
     - The "cannot determine" response includes the last FAIL issues so callers can see why.
     - Do NOT call normalize_answer with an unverified answer.
     - Do NOT call the verifier again.

   Count attempts by the number of verification attempt records in verification.txt.
   ```

3. **Update the existing verification.txt instruction** in SYSTEM_PROMPT — Remove the old stub line:
   ```
   After completing all calculation, WRITE to {uid}/verification.txt:
     Content: verification: pending (Phase 4)
   ```
   Replace it with:
   ```
   After completing all calculation, proceed to the Verification and Retry Protocol section below.
   verification.txt is managed by the verification protocol — do not write to it manually.
   ```

Keep all other SYSTEM_PROMPT sections unchanged. Keep `run_question` and `run_question_with_messages` unchanged.
