## Summary

This phase has three parallel tracks: (1) rewriting the system prompt with corpus-specific retrieval rules, (2) upgrading tool docstrings with corpus structure hints and parallel-call guidance, and (3) replacing the current FastAPI `/run` server with an A2A-protocol-compliant server based on the `a2a-sdk` library. All three are independent and can be planned as separate tasks.

The A2A integration is the highest-complexity track. The current server exposes a custom `/run` POST endpoint; the target architecture exposes a JSON-RPC endpoint at `/` (the A2A `message/send` method) and an agent card at `/.well-known/agent-card.json`, both served by `A2AStarletteApplication` from `a2a-sdk`. The existing `deepagents`-based agent logic stays unchanged — only the transport layer is replaced. The recommendation is **adapt, not rewrite**: wrap the existing `run_question()` in a new `AgentExecutor` class, replace `server.py`, and add a `Dockerfile`.

Prompt and tool improvements are low-risk edits to `agent.py` (SYSTEM_PROMPT string) and individual tool docstrings. The key prompt addition is a mandatory two-file retrieval rule: for any FY N question, always call `route_files` twice or with both year N and year N+1 before searching. Parallelizable tool calls are: (a) `route_files` for two year candidates simultaneously, and (b) `search_in_file` across multiple returned file paths simultaneously.

**Primary recommendation:** Implement all three tracks in sequence — prompts and docstrings first (low risk, fast), A2A server second (higher complexity), Docker last (build on working A2A server).

### Pattern 1: AgentExecutor Wrapping run_question()

**What:** The A2A SDK's `AgentExecutor.execute()` receives `context: RequestContext` and writes to `event_queue: EventQueue`. Extract the question text from the incoming A2A message, call `run_question(uid, question)` synchronously in a thread, and enqueue the result as an artifact.

**When to use:** Any time existing synchronous agent logic must be exposed via A2A.

**Example (derived from official a2a-sdk docs + LangGraph bridge pattern):**
```python
# Source: https://a2a-protocol.org/latest/tutorials/python/4-agent-executor/
#         https://a2aprotocol.ai/blog/a2a-langraph-tutorial-20250513
from a2a.server.agent_execution import AgentExecutor, RequestContext
from a2a.server.events import EventQueue
from a2a.utils import new_agent_text_message
import asyncio

class OfficeQAAgentExecutor(AgentExecutor):
    async def execute(self, context: RequestContext, event_queue: EventQueue) -> None:
        # Extract question and uid from incoming A2A message
        question = get_message_text(context.message)
        uid = context.task_id or str(uuid4())
        # Run synchronous agent in thread pool to avoid blocking event loop
        answer = await asyncio.to_thread(run_question, uid, question)
        await event_queue.enqueue_event(new_agent_text_message(answer))

    async def cancel(self, context: RequestContext, event_queue: EventQueue) -> None:
        raise UnsupportedOperationError("cancel not supported")
```

### Pattern 2: A2AStarletteApplication Server

**What:** Replace FastAPI app with `A2AStarletteApplication`. It auto-registers two routes: `/.well-known/agent-card.json` (GET) and `/` (POST JSON-RPC for `message/send` and `message/stream`).

**Example:**
```python
# Source: https://a2a-protocol.org/latest/tutorials/python/5-start-server/
from a2a.server.apps import A2AStarletteApplication
from a2a.server.request_handlers import DefaultRequestHandler
from a2a.server.tasks import InMemoryTaskStore
from a2a.types import AgentCapabilities, AgentCard, AgentSkill

skill = AgentSkill(
    id='officeqa',
    name='OfficeQA Finance Agent',
    description='Answers fiscal/financial questions from US Treasury bulletin corpus',
    tags=['finance', 'treasury', 'rag'],
    examples=['What was total defense spending in FY1940?'],
)

agent_card = AgentCard(
    name='OfficeQA Finance Agent',
    description='RAG agent over US Treasury bulletins, 1939-2025',
    url='http://0.0.0.0:9009/',
    version='1.0.0',
    default_input_modes=['text'],
    default_output_modes=['text'],
    capabilities=AgentCapabilities(streaming=False),
    skills=[skill],
)

request_handler = DefaultRequestHandler(
    agent_executor=OfficeQAAgentExecutor(),
    task_store=InMemoryTaskStore(),
)

server = A2AStarletteApplication(
    agent_card=agent_card,
    http_handler=request_handler,
)
uvicorn.run(server.build(), host='0.0.0.0', port=9009)
```

### Pattern 5: Parallel Tool Calls in System Prompt

**What:** Instruct the agent to issue multiple tool calls simultaneously rather than sequentially.

**Recommended parallelization points:**
1. `route_files` × 2 — call once for "FY N question" (which routes to year N files) and once with year N+1 explicitly referenced, OR instruct the agent to request both years in a single `route_files` call (the tool already handles multiple year references in a question).
2. `search_in_file` × N — once `route_files` returns a list of file paths, call `search_in_file` for all relevant files simultaneously rather than one at a time.

**Prompt instruction pattern (Anthropic best practice for parallel tool use):**
```
## Parallel Retrieval Rule
When route_files returns multiple file paths, call search_in_file for ALL
relevant files in a SINGLE turn (parallel calls), not one file per turn.
This applies whenever two or more files may contain the target data.
```

### Anti-Patterns to Avoid

- **Keeping `/run` as the only endpoint:** AgentBeats registration requires `/.well-known/agent-card.json` and A2A JSON-RPC at `/`. A custom `/run` will not satisfy platform requirements.
- **Creating asyncio primitives at module level in the new server:** The existing `server.py` already solves this correctly with lifespan; carry the pattern forward into the A2A server.
- **Duplicating "cannot determine" logic in system prompt:** The verifier already handles give-up after 3 attempts. Adding it to the system prompt creates conflicting policies.
- **Sequential search_in_file calls:** Route files returns 12+ candidate paths; searching them one at a time wastes turns and time.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Agent card endpoint | Custom `/agent.json` route | `A2AStarletteApplication` auto-registers it | Wrong URL (`/agent.json` vs `/.well-known/agent-card.json`) breaks discovery |
| JSON-RPC routing | Custom `@app.post("/message/send")` | `DefaultRequestHandler` | Protocol compliance, error codes, streaming, task lifecycle |
| Task state machine | Custom task tracking dict | `InMemoryTaskStore` | Correct terminal-state re-entry prevention |
| A2A message parsing | Custom JSON parsing | `get_message_text()` from a2a-sdk | Handles part types, content blocks |

**Key insight:** The a2a-sdk handles all protocol plumbing. The only custom code needed is `AgentExecutor.execute()` — the 10-line bridge between A2A context and `run_question()`.

## Common Pitfalls

### Pitfall 1: Port mismatch between agent card URL and server bind
**What goes wrong:** `AgentCard(url='http://localhost:9009/')` but server binds to port 8000. The agent card URL is what other agents use to discover this agent; it must match the actual running port.
**Why it happens:** Copying example code without updating port constants.
**How to avoid:** Define a single `PORT = 9009` constant and use it in both `AgentCard(url=...)` and `uvicorn.run(..., port=PORT)`.
**Warning signs:** Conformance test `test_agent_card` fails with connection refused on port in card URL.

### Pitfall 2: Blocking the async event loop in AgentExecutor
**What goes wrong:** Calling `run_question()` directly (synchronous) inside `async def execute()` blocks the entire event loop.
**Why it happens:** `run_question` runs a LangGraph agent synchronously; `execute()` is async.
**How to avoid:** Always wrap synchronous agent calls with `await asyncio.to_thread(run_question, uid, question)` — identical pattern to existing `server.py`.
**Warning signs:** Server accepts no new requests while one agent run is in progress.

### Pitfall 3: requirements.txt vs pyproject.toml in Docker
**What goes wrong:** RDI agent-template uses `uv sync --locked` which requires `pyproject.toml` + `uv.lock`. The current project uses `requirements.txt`.
**Why it happens:** Two different dependency management conventions.
**How to avoid:** Either (a) add `pyproject.toml` with same deps and generate `uv.lock`, OR (b) use a simpler Dockerfile with `pip install -r requirements.txt`. Option (b) is lower friction; option (a) aligns exactly with agent-template.
**Warning signs:** `uv sync` fails with "no pyproject.toml found".

### Pitfall 4: A2A SDK version drift
**What goes wrong:** `a2a-sdk` is at v0.3.25 as of 2026-03-21 with an active `1.0-dev` branch. API surface may differ between 0.3.x and future 1.0.
**Why it happens:** Fast-moving protocol spec.
**How to avoid:** Pin to `a2a-sdk[http-server]==0.3.25` or `>=0.3.20,<1.0` in `requirements.txt`. Do not use `1.0-dev` branch.

### Pitfall 5: FY N+1 retrieval rule not triggering for bare-year questions
**What goes wrong:** The `route_files` tool already maps FY N to calendar-year months Oct(N-1)–Sep(N). But the agent doesn't know to also look at bulletins from Oct(N)–Dec(N) (the first quarter of fiscal year N+1, often where end-of-year FY N summaries are published).
**Why it happens:** FY N summaries are often published retrospectively in the first bulletin of the following fiscal year.
**How to avoid:** System prompt must explicitly instruct: "For any FY N question, after calling route_files once for the question text, also call route_files with 'FY {N+1}' to get the following year's bulletins, then search both sets."

### Pitfall 6: Tool docstring changes don't affect already-compiled agent
**What goes wrong:** Tool docstrings are read at agent graph compile time (inside `create_agent()`). If docstrings are changed but a cached compiled graph is in use, old descriptions persist.
**Why it happens:** In the current architecture `create_agent()` is called fresh per question, so this is not a risk. But it's worth noting for future caching.
**How to avoid:** No action needed — `create_agent()` is already called once per question.

## Open Questions

1. **Does `deepagents.create_deep_agent` support streaming output?**
   - What we know: The current agent invokes synchronously via `.invoke()`. A2A supports streaming via `message/stream` endpoint.
   - What's unclear: Whether streaming the agent's intermediate steps over A2A is required by AgentBeats or if a single final artifact is sufficient.
   - Recommendation: Implement non-streaming first (`capabilities=AgentCapabilities(streaming=False)` in agent card). Streaming can be added in a follow-on phase if AgentBeats assessment requires it.

2. **Does the existing `/health` GET endpoint need to be preserved alongside the A2A server?**
   - What we know: `A2AStarletteApplication.build()` accepts `**kwargs` passed to Starlette constructor, and `.routes()` returns a list of Route objects that can be merged with custom routes.
   - What's unclear: Whether AgentBeats or the benchmark runner probes `/health`.
   - Recommendation: Add a `/health` route to the Starlette app manually if needed. The A2A SDK does not prevent adding extra routes.

3. **What `uid` to pass to `run_question()` from the A2A executor?**
   - What we know: The current server uses the `uid` field from the `RunRequest` body. The A2A `RequestContext` exposes `context.task_id` (the task ID assigned by the SDK) and the message's `messageId`.
   - What's unclear: Whether `task_id` is stable enough to use as the scratch directory key across retries.
   - Recommendation: Use `context.task_id` if available, else `context.message.messageId`. Document this in the executor.
