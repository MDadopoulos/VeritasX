**Test functions:**

1. **test_smoke_question_produces_six_scratch_files** (integration):
   - Pick question 1 (simple lookup)
   - Call `run_question(uid, question)`
   - Call `verify_scratch_complete(uid)`
   - Assert complete=True (all 6 files exist and non-empty)
   - Assert evidence.txt contains "Source:" (annotated span format)
   - Assert answer.txt has at least 2 lines (answer + rationale)
   - Clean up scratch dir after

2. **test_smoke_extracted_values_have_units** (integration):
   - Pick question 2 (calculation question — MUST be a question that requires numeric extraction)
   - Call `run_question(uid, question)`
   - Read `./scratch/{uid}/extracted_values.txt`
   - **Assert that extracted_values.txt is non-empty** (at least 1 non-blank line). If the file is empty, fail with a clear message: "extracted_values.txt is empty — the chosen question did not exercise numeric extraction. Pick a calculation question."
   - Assert each non-empty line matches pattern `\w+ = .+ \(.+\)` (name = value (unit))
   - This validates success criterion 2 and SCR-03
   - Clean up after

3. **test_smoke_rerun_idempotent** (integration):
   - Pick question 1 again
   - Call `run_question(uid, question)` — first run
   - Call `verify_scratch_complete(uid)` — assert complete=True
   - Assert no errors raised
   - Call `run_question(uid, question)` — second run (same UID, overwrites scratch)
   - Call `verify_scratch_complete(uid)` — assert complete=True again
   - Assert scratch dir exists (was recreated, not left missing)
   - Do NOT assert answer_1 == answer_2 — LLM responses are non-deterministic. Instead verify structural correctness: both runs produce complete scratch output without errors.
   - This validates success criterion 3 (idempotent re-run)
   - Clean up after

4. **test_smoke_verification_txt_is_stub** (integration):
   - After any smoke run, check verification.txt contains "pending" or "Phase 4"
   - This validates the stub requirement from research Open Question 3

5. **test_smoke_write_todos_before_retrieval** (integration):
   - Run a question and capture the full result from `run_question` (modify run_question call or access agent result directly — may need to return the full result dict, or add a helper that returns messages)
   - Alternatively: inspect `result["messages"]` for ToolMessage ordering
   - Assert that the first ToolMessage with tool name "write_todos" appears BEFORE any ToolMessage with tool name "route_files" or "search_in_file" in the messages list
   - If run_question only returns a string, create a helper `run_question_with_messages(uid, question)` that returns `{"answer": str, "messages": list}` — or refactor run_question to optionally return messages
   - This validates success criterion 5 (write-todos gate enforcement) via automation, not just human checkpoint
