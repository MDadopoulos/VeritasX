## Summary

Phase 5.2 is a pure refactor: no new retrieval or arithmetic capabilities. The three work streams are (1) swap `FilesystemBackend` for `LocalShellBackend` with env-var-driven root path, (2) redesign the search-agent so it writes evidence to scratch files instead of relaying data inline to the orchestrator, and (3) wire the three pre-authored SKILL.md bundles (quant-stats, cpi-inflation-adjuster, historical-fx) into the harness via `SkillsMiddleware`.

The current `harness.py` has two problems the phase fixes. First, the backend root path is hardcoded as an absolute Windows path (`C:/Users/mdado/...`), which breaks on any other machine. Second, the search-agent system prompt tells the agent to return compact findings inline rather than writing to the UID scratch directory â€” this populates the orchestrator's context window with raw BM25 data, exactly what the Claudie architectural principle warns against. Both problems have straightforward fixes given the existing library infrastructure.

Skills integration requires understanding that `SkillsMiddleware` is path-discovery-based: it calls `backend.ls(source_path)` and scans for subdirectories containing `SKILL.md`. The three skill bundles live at `skills/` in the project root, not inside `workspace/`. The harness must resolve that path at startup and provide it via a second `FilesystemBackend` or ensure the `LocalShellBackend` root covers it â€” or use a second dedicated skills backend.

**Primary recommendation:** Replace `FilesystemBackend` with `LocalShellBackend(root_dir=SCRATCH_DIR, virtual_mode=True, inherit_env=True)`, resolve `SCRATCH_DIR` from env with a `Path(__file__).parent.parent.parent / "agentspace/scratch"` default, create a separate `FilesystemBackend` rooted at the skills directory for `SkillsMiddleware`, and update the search-agent system prompt to write evidence files rather than return inline text.

### Pattern 1: Backend Initialization with Env-Var Root Path

**What:** Resolve `SCRATCH_DIR` at harness startup using env var with a computed default, then construct `LocalShellBackend` with `virtual_mode=True, inherit_env=True`.

**When to use:** Whenever the backend root path must be portable across machines and deployment environments.

**Example:**

```python
# Source: workspace/deepagents/deepagents/backends/local_shell.py (verified)
import os
from pathlib import Path
from deepagents.backends import LocalShellBackend

# Resolve scratch dir â€” same pattern as CORPUS_DIR in config.py
_PROJECT_ROOT = Path(__file__).parent.parent.parent  # workspace/../ = project root
_DEFAULT_SCRATCH = _PROJECT_ROOT / "agentspace" / "scratch"
SCRATCH_DIR = Path(os.environ.get("SCRATCH_DIR", str(_DEFAULT_SCRATCH)))

backend = LocalShellBackend(
    root_dir=SCRATCH_DIR,
    virtual_mode=True,
    inherit_env=True,
)
```

Note: `LocalShellBackend.__init__` emits a `DeprecationWarning` if `virtual_mode` is not specified explicitly. Always pass it.

### Pattern 2: Separate Skills Backend

**What:** Use a plain `FilesystemBackend` with `virtual_mode=True` rooted at the skills directory for `SkillsMiddleware`. Skills are read-only; they don't need `LocalShellBackend`'s execute capability.

**When to use:** When skills live in a different directory tree than scratch. Keeps skill discovery isolated from scratch I/O.

**Example:**

```python
# Source: workspace/deepagents/deepagents/middleware/skills.py (verified)
from deepagents.backends import FilesystemBackend
from deepagents.middleware.skills import SkillsMiddleware

_SKILLS_ROOT = Path(os.environ.get("SKILLS_DIR", str(_PROJECT_ROOT / "skills")))
skills_backend = FilesystemBackend(root_dir=_SKILLS_ROOT, virtual_mode=True)

skills_middleware = SkillsMiddleware(
    backend=skills_backend,
    sources=["/quant-stats/", "/cpi-inflation-adjuster/", "/historical-fx/"],
)
```

`SkillsMiddleware` calls `backend.ls(source_path)` to find subdirectories, then looks for `SKILL.md` in each. With `virtual_mode=True` on `FilesystemBackend`, paths like `/quant-stats/` map to `{skills_root}/quant-stats/`. The three SKILL.md files all have valid frontmatter and match their directory names â€” no validation issues expected.

### Pattern 3: Search-Agent File-Writing Pattern

**What:** Update `SEARCH_AGENT_SYSTEM_PROMPT` so the search-agent writes raw evidence to `{uid}/evidence.txt` and extracted values to `{uid}/extracted_values.txt` via the filesystem tools, then returns ONLY a compact pointer message.

**When to use:** Always â€” this is the Claudie principle: subagents dump to disk, orchestrator reads file references.

**Current state (to change):**
```python
# harness.py â€” current search agent return format (WRONG):
# "Return ONLY a compact findings summary."
# variable_name = value (unit), source: filename
```

**Target state:**

```python
SEARCH_AGENT_SYSTEM_PROMPT: str = """\
...

After EACH search_in_file call, APPEND to {uid}/evidence.txt:
  Source: {file_path}
  {span_text}
```

This means the orchestrator's system prompt must also change: it should no longer expect inline compact findings. Instead it reads from `{uid}/extracted_values.txt`.

### Pattern 4: UID Scratch Path Propagation to Search-Agent

**What:** The search-agent needs to know the UID in order to write to the correct `{uid}/` subdirectory.

**Current state:** The orchestrator's task call passes the description as a plain-English question. The UID is not part of the search-agent task description in the current prompt.

**Options evaluated:**

1. **Task description injection (recommended):** Orchestrator system prompt instructs it to include the UID in every `task()` call: `task(subagent_type='search-agent', description='UID: {uid} | Task: Find...')`. The search-agent system prompt shows how to parse the UID prefix.

2. **Env var:** Set `QUESTION_UID` env var before each question. Fragile in async concurrent execution.

3. **Subagent config field:** `SubAgent` TypedDict does not have a `context` field for per-invocation data beyond `description`. Not viable without harness changes.

**Recommendation:** Option 1 â€” task description injection. The orchestrator already uses `{uid}` in its scratch-writing instructions. Extending that to the search-agent task call is natural and requires no new infrastructure.

### Pattern 5: Scratch File Layout Evaluation

**Current 6-file structure:**
- `evidence.txt` â€” raw spans from search-agent (CHANGE: written by search-agent)
- `tables.txt` â€” raw table blocks (CHANGE: written by search-agent)
- `extracted_values.txt` â€” numeric values (CHANGE: written by search-agent)
- `calc.txt` â€” arithmetic trace (KEEP: written by orchestrator)
- `verification.txt` â€” verifier audit trail (KEEP: written by verifier subagent)
- `answer.txt` â€” final answer (KEEP: written by orchestrator)

**Recommendation:** Keep all 6 files. The split is logically clean: search-agent owns first 3, orchestrator owns calc.txt + answer.txt, verifier owns verification.txt. No restructuring needed.

### Anti-Patterns to Avoid

- **Hardcoded absolute paths in harness.py:** The current `FilesystemBackend(root_dir="C:/Users/mdado/...")` is the primary bug. Never hardcode OS-specific paths.
- **Sharing one backend between scratch and skills:** Skills are read-only and live outside the scratch tree. A shared backend means the skills path is exposed to the scratch writing agents.
- **Search-agent returning data inline:** The Claudie principle is the architectural cornerstone. Returning extracted values inline defeats context-window isolation.
- **Not passing `virtual_mode` explicitly to `LocalShellBackend`:** The library emits a `DeprecationWarning` if omitted; the default will change in 0.5.0.
- **Creating `agentspace/scratch` directory in `scratch.py` only:** The `agentspace/scratch` root must exist before `prepare_scratch(uid)` can create `{root}/{uid}/`. Either create it in harness startup, or update `prepare_scratch` to use `mkdir(parents=True)` on the root first. Current `scratch.py` only calls `scratch_dir.mkdir(parents=True)` on `SCRATCH_ROOT / uid` â€” that will create both levels if the root doesn't exist, so it is actually fine.

## Common Pitfalls

### Pitfall 1: Hardcoded Path Breaks on Non-Windows or Different User
**What goes wrong:** `FilesystemBackend(root_dir="C:/Users/mdado/Downloads/LucidOwlF/agentspace/scratch")` crashes on deployment or any other machine.
**Why it happens:** The path was set manually during development; no env var fallback.
**How to avoid:** Use `os.environ.get("SCRATCH_DIR", str(Path(__file__).parent.parent.parent / "agentspace" / "scratch"))` in harness.py. Mirror the `CORPUS_DIR` pattern from `config.py`.
**Warning signs:** `FileNotFoundError` on first run on a new machine; works locally but fails in Docker.

### Pitfall 2: SkillsMiddleware Can't Find SKILL.md Files
**What goes wrong:** `SkillsMiddleware` scans `backend.ls(source_path)` but finds no subdirectories, or subdirectories with no `SKILL.md`. Skills silently not loaded.
**Why it happens:** The `source_path` passed to `SkillsMiddleware` must be a path relative to the backend's `root_dir`. If `root_dir` is the skills directory, then `/quant-stats/` is correct. If `root_dir` is the project root, then `/skills/quant-stats/` is needed.
**How to avoid:** Use a dedicated `FilesystemBackend(root_dir=SKILLS_ROOT)` and pass `/quant-stats/`, `/cpi-inflation-adjuster/`, `/historical-fx/` as sources. Verify: `backend.ls("/")` should show those three directories.
**Warning signs:** No skills appear in the system prompt; `SkillsMiddleware` logs `"No skills available yet"`.

### Pitfall 3: Search-Agent Writes to Wrong UID Directory
**What goes wrong:** Search-agent writes evidence to `{uid}/evidence.txt` but uses the wrong UID (from a prior question or hardcoded default).
**Why it happens:** The UID is only known at question invocation time, not at harness build time. The search-agent must receive the UID via the task description.
**How to avoid:** Orchestrator system prompt must explicitly instruct: "Every call to the search-agent MUST include 'UID: {uid}' at the start of the description field." Search-agent system prompt must show how to parse and use this prefix.
**Warning signs:** Evidence files appearing in the wrong UID directory; verifier failing because it reads the wrong evidence.

### Pitfall 4: LocalShellBackend virtual_mode DeprecationWarning Becomes Error
**What goes wrong:** If deepagents upgrades to 0.5.0, `virtual_mode` defaults change; omitting it may raise an error rather than a warning.
**Why it happens:** The library docstring says "virtual_mode default will change in deepagents 0.5.0." The local copy is pre-0.5.0 but this matters if the library is updated.
**How to avoid:** Always specify `virtual_mode=True` explicitly. This is already in the locked decision.
**Warning signs:** `DeprecationWarning` in current logs from `LocalShellBackend` if not specified.

### Pitfall 5: Skills Backend and Scratch Backend Sharing State
**What goes wrong:** If a single backend is used for both skills and scratch, agents may accidentally write to the skills directory or the skills source path scan picks up scratch files.
**Why it happens:** `SkillsMiddleware` calls `backend.ls(source_path)` on the same backend that `FilesystemMiddleware` uses for scratch writes. If both share the same root, skills paths and scratch paths collide.
**How to avoid:** Two separate backend instances: `LocalShellBackend(root_dir=SCRATCH_DIR)` for scratch, `FilesystemBackend(root_dir=SKILLS_DIR)` for skills. Pass the correct backend to each middleware.
**Warning signs:** Skills listed with UIDs as names; scratch files being treated as skills.

### Pitfall 6: scratch.py SCRATCH_ROOT Diverges from harness.py Backend Root
**What goes wrong:** `scratch.py` uses `Path(__file__).parent.parent / "scratch"` (resolves to `workspace/scratch`) but harness.py uses `agentspace/scratch`. The scratch lifecycle helpers create directories in the wrong location.
**Why it happens:** `scratch.py` was written before the `agentspace/scratch` convention was decided. The two paths are currently inconsistent.
**How to avoid:** Update `scratch.py` `SCRATCH_ROOT` to read `SCRATCH_DIR` env var with default `project_root / "agentspace" / "scratch"`. The harness's backend root and `scratch.py`'s `SCRATCH_ROOT` must point to the same directory.
**Warning signs:** `prepare_scratch(uid)` creates the directory but the agent writes files to a different path; `verify_scratch_complete` always returns `missing: [all 6 files]`.
