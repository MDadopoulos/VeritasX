<decisions>
## Implementation Decisions

### Working Directory

- Root path default: `./agentspace/scratch` relative to project root (resolved at harness startup via `__file__` or known root)
- `SCRATCH_DIR` env var overrides the default — same pattern as `CORPUS_DIR`
- `FilesystemBackend` should read `SCRATCH_DIR` from env, fall back to project-relative default
- Per-UID directory structure stays (one folder per question UID)
- Internal file layout (evidence.txt, tables.txt, etc.) — Claude's discretion (evaluate and restructure if beneficial)

### Search-Agent File-Writing Pattern

- **Key architectural change:** search-agent has `FilesystemMiddleware` write access to the UID scratch directory
- Search-agent dumps raw evidence, table blocks, and extracted values directly into scratch files — does NOT relay data back inline
- Orchestrator receives a compact pointer (e.g., "findings written to evidence.txt") rather than raw BM25 spans
- This preserves orchestrator context window — orchestrator reasons from file pointers, not data summaries
- Follows Claudie principle: "subagents dump to local files; orchestrator directs subagents to files without engaging with data itself"

### Verifier Subagent

- Keep as-is: PASS/FAIL + token behavior is solid
- No changes to verifier in this phase

### Skills Integration

- Skills format: SKILL.md descriptors + Python implementations (full DeepAgents skill system)
- SKILL.md files already written by user; Python implementations ready to be added to workspace during phase execution
- Skills to wire: statistical calculations, calculation+verification flow, historical FX conversion, CPI/inflation adjuster
- Phase 5.2 responsibility: discover and register skill files in the harness at startup
- Skills directory location: to be determined when user drops files in — planner should accommodate wherever they land in workspace/

### Harness Architecture Principles

- Architecture inspired by harness-anatomy and Claudie guidelines: orchestrator-worker pattern with strict job separation
- Orchestrator = reasoning + delegation only; search-agent = retrieval + filesystem writes
- Tool access control: soft enforcement via system prompt only (hard block not needed for benchmark async context)
- System prompt: will be reviewed and improved during planning — existing prompt is a starting point (Claude's discretion for specifics)
- Memory model: per-UID filesystem is sufficient; no cross-question progress file needed (HTTP idempotency layer handles resumption)
- Execution context: async benchmark run — no interactive/HITL requirements

### Claude's Discretion

- Whether and how to restructure internal scratch file layout (evaluate current 6-file structure)
- Exact SKILL.md discovery/registration mechanism in harness
- System prompt improvements for orchestrator (keep planning gate; assess what else to tighten)
- How search-agent receives the UID scratch path (env var, task description, or subagent config)

</decisions>
