**Gathered:** 2026-03-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the opaque `create_deep_agent()` call in `agent.py` with a custom harness in `workspace/src/harness.py` that imports deepagents middleware primitives directly, assembles a custom middleware stack, and calls `langchain.agents.create_agent()` at the bottom. The harness introduces a dedicated search subagent that absorbs corpus retrieval + extraction, keeping the orchestrator's context window clean.

No new tools. No new agent capabilities. This is an architectural refactor of how the agent is assembled and how context flows.

</domain>

<decisions>
## Implementation Decisions

### Graph customization depth
- **Option 3 (Hybrid)**: Fork the middleware assembly logic, keep `create_agent()` at the bottom
- Do NOT build a custom `StateGraph` from scratch — `create_agent()` from `langchain.agents` handles graph compilation and is a well-maintained abstraction worth keeping
- Custom state fields (uid, retrieval_count on graph state) are NOT needed — scratch directory + MemorySaver per question is sufficient
- Reason: middleware control is enough for the OfficeQA use case; full graph ownership is unnecessary complexity

### Code location and update strategy
- New file: `workspace/src/harness.py` — owns middleware assembly and `create_agent()` call
- `workspace/deepagents/` source stays **unmodified** — deepagents is treated as a library
- `agent.py` keeps `run_question`, `run_question_with_messages`, and `SYSTEM_PROMPT`; `create_agent()` in agent.py is replaced by a call to `harness.create_harness_agent()`
- This separation allows `pip install --upgrade deepagents` without losing local customizations

### What harness.py removes vs keeps
- **Remove**: deepagents auto-registers `FilesystemMiddleware` file tools (ls, read_file, write_file, execute) on its internal general-purpose subagent — the harness controls which agents get which tools explicitly
- **Keep on orchestrator**: `TodoListMiddleware`, `FilesystemMiddleware` (orchestrator writes scratch files), `SubAgentMiddleware`, `SummarizationMiddleware`
- **Keep on search subagent**: `TodoListMiddleware`, `FilesystemMiddleware`, `SummarizationMiddleware`
- No custom middleware added — standard deepagents stack is sufficient

### Context isolation strategy
- The real problem is **context window pollution**: `search_in_file` returns large 20-line spans; multiple files × multiple searches fills the orchestrator's message history with raw corpus text
- Solution: search subagent absorbs all corpus text internally; orchestrator only receives compact findings
- Existing per-question isolation (fresh `MemorySaver` + `prepare_scratch(uid)` per `run_question` call) is retained as-is — no changes needed there

### Search subagent design
- **One search subagent** (not multiple specialized agents)
- **Owns**: `route_files` + `search_in_file` + `extract_table_block` — full retrieval + extraction pipeline
- **Does NOT own**: `calculate`, `pct_change`, `sum_values`, `normalize_answer` — those stay on the orchestrator
- **Instructed via natural language task**: orchestrator calls the subagent with a plain-English task like "Find defense expenditures for FY1940 across all relevant files"
- **Returns**: a compact findings summary + extracted numeric values (e.g. `defense_1940 = 2602 (millions), source: bulletin_1940_03.txt`) — no raw corpus text returned to orchestrator
- FY Adjacency Rule and Parallel Search Rule should be baked into the search subagent's system prompt (move them out of the main SYSTEM_PROMPT)

### Summarization middleware
- `SummarizationMiddleware` on **both** orchestrator and search subagent
- The search subagent accumulates large corpus text and especially needs it
- Orchestrator benefits for long multi-file questions

### TodoListMiddleware
- On **both** orchestrator and search subagent
- Orchestrator plans the full question-answering sequence
- Subagent plans its search sequence (which files to search, in what order)

### Claude's Discretion
- Exact system prompt for the search subagent (tone, instruction format)
- Whether to give the search subagent a name like "retrieval-agent" or "search-agent" for SubAgent spec
- Handling of the `PatchToolCallsMiddleware` — include if needed for model compatibility

</decisions>

<specifics>
## Specific Ideas

- The user referenced https://blog.langchain.com/the-anatomy-of-an-agent-harness/ — the key takeaway applied here: the harness controls context flow, and the main design decision is ensuring raw corpus text never lands in the orchestrator's context window
- The deepagents source is vendored locally at `workspace/deepagents/` — useful as a reference for middleware APIs, but should not be modified
- The `graph.py` file in deepagents is the reference for how `create_deep_agent` assembles its middleware stack — `harness.py` is essentially a purpose-built version of that assembly for OfficeQA
