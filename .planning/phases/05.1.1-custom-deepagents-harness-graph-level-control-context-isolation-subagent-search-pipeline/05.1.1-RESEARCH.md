## Summary

The harness refactor replaces the single `create_deep_agent()` call in `agent.py` with a purpose-built middleware assembly in `harness.py`. The deepagents library's `graph.py` is the reference: it shows exactly which middleware objects get instantiated for the main agent and for each subagent, and how `create_agent()` from `langchain.agents` is called at the end.

The core task is to reproduce the middleware assembly logic from `graph.py` in `harness.py`, but with deliberate control over which agents see which tools. The search subagent receives only the retrieval tools (`route_files`, `search_in_file`, `extract_table_block`); the orchestrator receives only the calculation and normalization tools (`calculate`, `pct_change`, `sum_values`, `normalize_answer`). This split ensures raw corpus text â€” large 20-line BM25 spans from `search_in_file` â€” is absorbed inside the subagent's context window rather than the orchestrator's.

The `SubAgentMiddleware` new API (introduced in deepagents, distinct from the deprecated legacy API) requires fully-specified `SubAgent` TypedDicts that include `name`, `description`, `system_prompt`, `model`, `tools`, and `middleware`. The middleware list passed into each `SubAgent` spec is treated as the complete middleware stack (no defaults injected by `SubAgentMiddleware` in new-API mode) â€” the caller is responsible for building the full middleware list.

**Primary recommendation:** Reproduce `graph.py`'s middleware assembly pattern in `harness.py`; use the new `SubAgentMiddleware` API (pass `backend=` and fully-specified `subagents=`); build the search subagent's middleware stack manually following the same order used in `graph.py` for subagents.

## Architecture Patterns

### How `graph.py` Assembles a Subagent (Verified Pattern)

When `create_deep_agent()` processes a `SubAgent` spec, it builds the subagent middleware in this exact order (source: `workspace/deepagents/deepagents/graph.py` lines 244â€“254):

```python
subagent_middleware = [
    TodoListMiddleware(),
    FilesystemMiddleware(backend=backend),
    create_summarization_middleware(subagent_model, backend),
    PatchToolCallsMiddleware(),
]
# optional: SkillsMiddleware if spec has 'skills'
# extend with spec.get("middleware", [])
subagent_middleware.append(AnthropicPromptCachingMiddleware(unsupported_model_behavior="ignore"))
```

The `SubAgent` spec is then: `{**spec, "model": resolved_model, "tools": spec_tools, "middleware": subagent_middleware}`.

`harness.py` must replicate this pattern for the search subagent.

### SubAgentMiddleware New API vs Deprecated API

`SubAgentMiddleware` has two distinct APIs:

**New API (use this):** Pass `backend=` and `subagents=`. Each `SubAgent` in the list **must** have `model` and `tools` specified. The `middleware` in the spec is used as-is (no defaults injected). Source: `subagents.py` lines 636â€“668.

```python
SubAgentMiddleware(
    backend=backend,
    subagents=[
        {
            "name": "search-agent",
            "description": "...",
            "system_prompt": "...",
            "model": model,
            "tools": [route_files, search_in_file, extract_table_block],
            "middleware": [
                TodoListMiddleware(),
                FilesystemMiddleware(backend=backend),
                create_summarization_middleware(model, backend),
                PatchToolCallsMiddleware(),
                AnthropicPromptCachingMiddleware(unsupported_model_behavior="ignore"),
            ],
        }
    ],
)
```

**Deprecated API (do not use):** Passes `default_model`, `default_tools`, etc. Triggers `DeprecationWarning`. Scheduled for removal in version 0.5.0.

### System Prompt Combination

`graph.py` appends `BASE_AGENT_PROMPT` to any custom `system_prompt` (source: `graph.py` lines 303â€“310):

```python
# String case:
final_system_prompt = system_prompt + "\n\n" + BASE_AGENT_PROMPT
```

`harness.py` must do the same when calling `create_agent()` directly, because `create_agent()` from `langchain.agents` does **not** append `BASE_AGENT_PROMPT` automatically â€” that is done by `create_deep_agent()`. If `harness.py` calls `create_agent()` directly without appending `BASE_AGENT_PROMPT`, the orchestrator loses the deepagents base behavior instructions.

### Subagent State Passing

When the orchestrator calls `task(subagent_type="search-agent", description="...")`, the `SubAgentMiddleware` passes the current orchestrator state to the subagent (minus excluded keys: `messages`, `todos`, `structured_response`, `skills_metadata`, `memory_contents`) and sets the messages to `[HumanMessage(content=description)]`. The subagent's final message is returned as a `ToolMessage` to the orchestrator. Raw intermediate messages are not visible to the orchestrator. This is the isolation mechanism. Source: `subagents.py` lines 422â€“428 and 402â€“420.

### General-Purpose Subagent Suppression

`create_deep_agent()` automatically inserts a `general-purpose` subagent if none is named that in the subagents list (line 266). When `harness.py` uses `SubAgentMiddleware` new API directly, this auto-insertion does **not** happen â€” `SubAgentMiddleware` only adds the subagents it is explicitly given. The harness therefore has full control over which subagents the orchestrator can call.

### Recommended Structure for `harness.py`

```
workspace/src/harness.py

def create_harness_agent() -> CompiledStateGraph:
    1. Import model, backend, tools
    2. Build search subagent middleware stack (ordered like graph.py)
    3. Build search_subagent SubAgent TypedDict
    4. Build orchestrator middleware stack:
       [TodoListMiddleware, FilesystemMiddleware, SubAgentMiddleware(search_subagent),
        create_summarization_middleware, PatchToolCallsMiddleware,
        AnthropicPromptCachingMiddleware]
    5. Combine ORCHESTRATOR_SYSTEM_PROMPT + "\n\n" + BASE_AGENT_PROMPT
    6. Call create_agent(model, tools=orchestrator_tools, middleware=...,
                        system_prompt=..., checkpointer=MemorySaver())
    7. Return compiled agent
```

## Common Pitfalls

### Pitfall 1: Forgetting BASE_AGENT_PROMPT in harness.py
**What goes wrong:** `harness.py` calls `create_agent()` directly. Unlike `create_deep_agent()`, `create_agent()` does NOT append `BASE_AGENT_PROMPT`. The orchestrator loses its core behavior instructions (conciseness rules, verification loop, task completion behavior).
**Why it happens:** `graph.py` handles the concatenation transparently; bypassing it drops the concatenation.
**How to avoid:** In `harness.py`, explicitly do `system_prompt = ORCHESTRATOR_SYSTEM_PROMPT + "\n\n" + BASE_AGENT_PROMPT` before the `create_agent()` call.
**Warning sign:** Orchestrator is verbose, adds unnecessary preamble, stops mid-task.

### Pitfall 2: SubAgentMiddleware New API Requires model + tools in SubAgent Spec
**What goes wrong:** Passing a `SubAgent` dict without `model` or `tools` to `SubAgentMiddleware(backend=..., subagents=[...])` raises `ValueError: SubAgent 'X' must specify 'model'`.
**Why it happens:** New API does not inject defaults; legacy API did. Source: `subagents.py` lines 637â€“641.
**How to avoid:** Always include `model` and `tools` in every `SubAgent` dict when using the new API.

### Pitfall 3: Search Subagent Middleware Stack Missing AnthropicPromptCachingMiddleware
**What goes wrong:** `AnthropicPromptCachingMiddleware` is missing from the subagent's middleware list. No error is raised, but prompt caching is disabled for the subagent, increasing latency and cost on Claude models.
**Why it happens:** Easy to forget; it's appended last and not obviously necessary.
**How to avoid:** Follow the exact order from `graph.py`: always append `AnthropicPromptCachingMiddleware(unsupported_model_behavior="ignore")` as the last item in any middleware list for any agent/subagent.

### Pitfall 4: Orchestrator Receives extract_table_block
**What goes wrong:** `extract_table_block` is left on the orchestrator's tool list. The orchestrator calls it directly, which means raw corpus span text passes through the orchestrator's context window before table extraction â€” defeating the isolation goal.
**Why it happens:** Old `agent.py` had `extract_table_block` commented out but present; easy to include accidentally.
**How to avoid:** Orchestrator tool list is: `[calculate, pct_change, sum_values, normalize_answer]` only. The search subagent owns `[route_files, search_in_file, extract_table_block]`.

### Pitfall 5: FY Adjacency Rule Left in Orchestrator System Prompt
**What goes wrong:** FY Adjacency Rule and Parallel Search Rule remain in `SYSTEM_PROMPT` in `agent.py`. The orchestrator follows them directly by calling retrieval tools, bypassing the search subagent.
**Why it happens:** Forgotten during the refactor; the rules were written to drive orchestrator tool use.
**How to avoid:** After creating `harness.py`, edit `SYSTEM_PROMPT` in `agent.py` to remove both rules. Replicate them in the search subagent's `system_prompt` in `harness.py`.

### Pitfall 6: PatchToolCallsMiddleware and Non-Anthropic Models
**What goes wrong:** `PatchToolCallsMiddleware` is kept but the model is a Gemini variant (`ChatGoogleGenerativeAI`). The middleware does nothing harmful for non-Anthropic models, but `AnthropicPromptCachingMiddleware(unsupported_model_behavior="ignore")` is the critical one â€” it silently skips caching for non-Anthropic models because of the `ignore` flag.
**Why it happens:** Code is model-agnostic; the project uses `get_model()` which can return Gemini or Claude.
**How to avoid:** Both `PatchToolCallsMiddleware` and `AnthropicPromptCachingMiddleware(unsupported_model_behavior="ignore")` are safe to include unconditionally. The `ignore` flag handles non-Anthropic models without error.

## SYSTEM_PROMPT Split Plan

### What stays in `agent.py`'s `SYSTEM_PROMPT` (orchestrator)

- Mandatory Planning Gate (write_todos before ANY tool call)
- Scratch File Writing Instructions (evidence.txt, extracted_values.txt, calc.txt, answer.txt)
- Tool Usage Rules (never inline arithmetic, always pct_change)
- Instructions for calling the search subagent and what to expect back

### What moves to the search subagent's `system_prompt` in `harness.py`

- Fiscal Year Adjacency Rule
- Parallel Search Rule
- Instructions for writing evidence to scratch files (subagent writes its own evidence.txt entries)
- Instructions for the compact return format: `variable_name = value (unit), source: file`

### New orchestrator instruction (replaces retrieval tool calls)

The orchestrator's `SYSTEM_PROMPT` should include guidance like:
- "To retrieve data, call `task` with `subagent_type='search-agent'` and a natural language description"
- "The search agent returns compact findings: `variable_name = value (unit), source: file`"
- "Do NOT call `route_files`, `search_in_file`, or `extract_table_block` directly"

## Open Questions

1. **`create_agent()` recursion_limit for subagent**
   - What we know: `create_deep_agent()` calls `.with_config({"recursion_limit": 1000})` on the compiled agent. Subagents in the new API are compiled via `create_agent()` in `_get_subagents()`, without `.with_config(recursion_limit=...)`.
   - What's unclear: Whether the search subagent needs its own recursion_limit config, or inherits from the parent graph's config.
   - Recommendation: For safety, call `.with_config({"recursion_limit": 200})` when building a `CompiledSubAgent` via the `runnable` key, OR accept the default (which is LangGraph's default of 25 â€” potentially too low for multi-file search). If using `SubAgent` TypedDict (not `CompiledSubAgent`), subagent compilation happens inside `SubAgentMiddleware._get_subagents()` which does not set recursion_limit. If the search subagent needs to search many files in parallel, this could be a problem. **Mitigation**: Use `CompiledSubAgent` to pre-compile the search agent with `.with_config({"recursion_limit": 200})`.

2. **`AnthropicPromptCachingMiddleware` on Gemini models**
   - What we know: `unsupported_model_behavior="ignore"` silently skips caching for non-Anthropic models.
   - What's unclear: Whether this adds measurable overhead for Gemini (the default model).
   - Recommendation: Include it unconditionally; the overhead is a no-op skip check per model call, negligible.

3. **Search subagent name: "search-agent" vs "retrieval-agent"**
   - What we know: The `SubAgent` `name` field is used as the key in `task(subagent_type=...)` calls. The orchestrator's system prompt should tell it which name to use.
   - What's unclear: Claude's discretion; no functional difference.
   - Recommendation: Use `"search-agent"` â€” shorter, clearer in tool call logs, and aligns with the phase description.
